### Finding #1: Path traversal in safe_filename/storage_path leads to arbitrary file write in host apps

| Field | Value |
|---|---|
| Severity | High |
| Status | enriched |
| CWE | CWE-22 |
| Location | `lib/upload_guard_workshop.rb:60-65` |
| Sinks | S1, S2, S8 |
| Affected | UploadGuard::Guard#safe_filename, UploadGuard::Guard#storage_path |

#### Source

```
        errors: errors,
        filename: safe_filename(upload)
      )
    end

    def safe_filename(upload)
      original_filename(upload).to_s.tr("\u0000", "").strip.gsub(/\s+/, "_")
    end

    def storage_path(base_dir, upload)
      File.join(base_dir, safe_filename(upload))
```

#### Trace

`storage_path(base_dir, upload)` (lib:64) calls `File.join(base_dir, safe_filename(upload))`. `safe_filename` (lib:60) calls `original_filename(upload).to_s.tr("\u0000", "").strip.gsub(/\s+/, "_")` — it removes NUL bytes and collapses whitespace but does **not** strip `../`, leading slashes, or absolute paths. `original_filename` (lib:94) returns `upload.original_filename` verbatim. The upload object comes from the host app's `params` (README.md:66, examples/rails_controller.rb:7 `params.require(:company).fetch(:logo)`), so `original_filename` is attacker-controlled multipart metadata. `File.join` performs no normalization, so `../` survives into the returned path. The host then writes to that path: examples/rails_controller.rb:35 `File.binwrite(path, upload.read)` (S8).

#### Trust boundary

Crosses boundary #1 (HTTP upload client, trusted:no). threat_model entry_points[3] (`upload.original_filename`, source lib:94, attacker_controllable:yes) and entry_points[4] (`safe_filename`, source lib:60) both document the value as attacker-controlled, with `caller_must_enforce` noting the gem does not strip traversal sequences. `base_dir` is the trusted operator value (entry_points[5]); only the filename crosses the boundary. The precondition (a host that writes to storage_path output) is not circular — the gem ships that exact usage in examples/rails_controller.rb, and the method name `safe_filename` invites callers to trust the output.

### Finding #3: Extension allowlist uses substring matching, accepting double-extension payloads

| Field | Value |
|---|---|
| Severity | Medium |

| CWE | CWE-434 |
| Location | `lib/upload_guard_workshop.rb:82-88` |
| Sinks | S4 |
| Affected | UploadGuard::Guard#extension_allowed? |

#### Source

```

    def content_type_allowed?(upload)
      allowed_mime_types.include?(upload.content_type.to_s)
    end

    def extension_allowed?(upload)
      return true if allowed_extensions.empty?

      filename = original_filename(upload).to_s.downcase
      allowed_extensions.any? { |extension| filename.index(extension) }
```

#### Trace

`validate` (lib:50) rejects unless `extension_allowed?(upload)` (lib:82) returns true. That method lowercases `original_filename` and returns `allowed_extensions.any? { |extension| filename.index(extension) }` (lib:87) — `String#index` is a substring search anywhere in the name, not a final-extension comparison. So `invoice.pdf.exe` matches `.pdf` (found at offset 7) even though the real extension is `.exe`. `original_filename` (lib:94) is attacker-controlled multipart metadata.

#### Trust boundary

Crosses boundary #1 (HTTP upload client, trusted:no). threat_model entry_points[2] (`upload.original_filename`, attacker_controllable:yes).

### Finding #2: Content-type allowlist and image?/document? predicates trust client-supplied metadata

| Field | Value |
|---|---|
| Severity | Medium |
| Status | new |
| CWE | CWE-434 |
| Location | `lib/upload_guard_workshop.rb:68-79` |
| Sinks | S3, S5, S6 |
| Affected | UploadGuard::Guard#content_type_allowed?, #image?, #document? |

#### Source

```

    def storage_path(base_dir, upload)
      File.join(base_dir, safe_filename(upload))
    end

    def image?(upload)
      upload.content_type.to_s.start_with?("image/")
    end

    def document?(upload)
      upload.content_type.to_s == "application/pdf"
```

#### Trace

`validate` (lib:49) rejects unless `content_type_allowed?(upload)` (lib:78) returns true, which is `allowed_mime_types.include?(upload.content_type.to_s)`. `upload.content_type` is the multipart `Content-Type` header the client sets, never the actual file bytes. Same for the public predicates `image?` (lib:68, `content_type.to_s.start_with?("image/")`) and `document?` (lib:72, `== "application/pdf"`). A caller that treats an accepted upload — or a true `image?`/`document?` — as proof of real file type is deceived.

#### Trust boundary

Crosses boundary #1 (HTTP upload client, trusted:no). threat_model entry_points[1] (`upload.content_type`, source lib:79, attacker_controllable:yes, caller_must_enforce: "gem trusts this verbatim; callers relying on accurate type detection must inspect file bytes themselves") and entry_points[6]/[7] for image?/document?. The gem provides content-type allowlisting only 'against declared metadata' (threat_model properties_provided[1], conditions: 'Only checks the content_type attribute, not the actual file bytes').
